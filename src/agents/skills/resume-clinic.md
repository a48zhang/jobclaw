# 简历诊断与改写 SOP (Resume Clinic)

### 场景
当用户要求“评价简历”“帮我改简历”“按某个岗位优化简历”，或用户刚上传 PDF 简历后要求分析时，执行本 SOP。

### 步骤
1. **读取材料**：
   - 优先检查简历是否存在；如果存在，使用 `read_pdf` 提取文本。
   - 同时读取 `data/userinfo.md`。
   - 如果 `data/resume.typ` 存在，也读取它。
2. **确定信息**：
   - 若用户已给出岗位名称、JD 或职位链接，按该岗位审查。
   - 若没有且岗位信息会显著影响判断，则使用 `request` 工具询问目标岗位或 JD；否则先给通用审查并说明可补充岗位再细化。
   - 推断用户当前状态(实习|校招|社招),避免提出与用户状态不符合的建议
3. **先做审查，不要直接改文件**：
   输出至少包含以下部分：
   - 红旗问题
   - 岗位匹配度
   - 量化指标缺口
   - 容易被追问的点
   - 改写策略
   输出前需要检查所有建议是否合理:
   - 建议是否与用户信息相符?比如用户是实习或校招,就不应该提出"添加生产环境经验"
   - 建议是否与真实世界相符?你的训练数据很可能已经过时,如果涉及时间矛盾:必须遵循当前system prompt提出的系统时间确定问题是否真实存在,避免训练数据过时造成的误报
4. **信息不足时追问**：
   - 对关键量化指标、职责范围、技术栈等缺口，优先用 `request` 工具向用户追问。
5. **用户要求改写时落地**：
   - 只有用户要求改了再继续,否则到此结束
   - 若存在 `data/resume.typ`，直接修改它。
   - 若只有上传的 PDF，没有 `resume.typ`，先基于已有信息生成新的 `data/resume.typ` 草稿，再继续修改。
   - 改写完成后调用 `typst_compile`，输入固定为 `data/resume.typ`。
   - 输出路径固定为 `output/resume.pdf`。
6. **输出前再次反思自己的输出是否符合注意事项的要求**
   - 当你结束时,首先复述下面的内容
   - 然后检查每一条建议是否违反了规则

### 注意事项
- 上传 PDF 时，必须区分“从 PDF 提取到的文本”和“基于文本做的推断”。
- `read_pdf` 解析失败时，必须明确说明失败原因；不能假装已经读懂 PDF。
- 对缺失的量化信息，优先向用户追问，不要虚构。
- 改写建议必须具体到句子级别，不能只说“多量化”“更突出亮点”。
- 如果涉及时间矛盾:必须遵循当前system prompt提出的系统时间确定问题是否真实存在,避免训练数据过时造成的误报
- 明确说明改了哪些段落。
- 不得编造经历、指标、学校、公司、项目结果。
- **必须遵循当前的系统时间确定问题**
