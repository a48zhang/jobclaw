# JobClaw 用户视角逻辑问题与修复方案

> 审查日期：2026-03-27  
> 审查范围：当前整个仓库的主用户链路，包括启动、初始化、聊天、职位看板、简历工具与 Web 控制台。

## 1. 审查目标

本文档不从代码风格或架构纯度出发，而是从真实用户的视角检查以下问题：

- 用户以为系统会怎么工作
- 系统实际上怎么工作
- 两者之间是否存在逻辑断裂、误导、状态不一致或高概率踩坑点

重点关注“主路径能否闭环”，而不是边缘技术细节。

---

## 2. 总体结论

当前版本已经具备一个可演示的产品骨架，但存在 3 条核心用户路径互相冲突的问题：

1. 初始化引导与服务启动前置条件冲突，导致新用户无法按 UI 引导完成首次配置。
2. 简历工具将任务提交给临时 Agent，但 Web 界面只完整观察主 Agent，导致“任务已提交”却看不到结果。
3. 职位看板的批量操作采用整表覆盖写回，与 Agent 的增量写入机制冲突，容易造成用户感知上的数据丢失。

如果不优先修复这些问题，用户会得到以下直接印象：

- “产品提示我这样做，但实际上根本做不到”
- “我点击了按钮，但结果没有回来”
- “我刚才看到的职位怎么突然没了”

这会直接伤害产品可信度。

---

## 3. 详细问题与修复方案

### 问题一：首次使用流程断裂

#### 3.1 用户视角表现

README 和首页引导都在表达同一个心智模型：

- 启动 Web 控制台
- 先填写 `targets.md` 和 `userinfo.md`
- 然后开始搜索职位

但实际服务启动前会强制校验：

- `API_KEY`
- `MODEL_ID`
- `BASE_URL`

只要这三项没配，服务就直接启动失败，用户连页面都看不到。

与此同时，Web 配置页只允许编辑：

- `targets.md`
- `userinfo.md`

并不提供 `config.json` 编辑能力。也就是说，产品引导用户通过 Web 完成初始化，但真正阻塞启动的配置又不在 Web 可配置范围内。

#### 3.2 影响

- 新用户首次体验直接中断。
- README、首页提示、后端行为三者不一致。
- 用户可能误以为产品坏了，而不是“还没配置模型”。

#### 3.3 根因

- 把“运行时配置编辑”与“启动前必需配置”混在一个 onboarding 叙事里。
- UI 只开放了业务侧 markdown 配置，没有开放基础运行配置。

#### 3.4 修复目标

让首次使用路径满足下面两种模式中的一种，并且只保留一种清晰叙事：

#### 方案 A：允许无模型配置启动 Web，进入引导模式

这是更推荐的方案。

实施方式：

1. 调整 `validateEnv()` 行为：
   - Web 模式下如果缺少模型配置，不直接抛异常。
   - 改为返回“配置不完整”的状态对象。
2. `runServer()` 在配置不完整时仍启动 Web 服务。
3. Web 首页展示明确的首次配置引导面板：
   - 缺少哪些字段
   - 如何填写
   - 当前功能哪些不可用
4. 新增 `config.json` 的只读展示或可编辑入口。
5. 聊天输入区、简历按钮、搜索动作在未完成基础配置时禁用，并给出原因。

优点：

- 用户可以真正通过 Web 完成初始化。
- 产品叙事统一。
- 错误从“启动失败”变为“页面内可理解的待配置状态”。


#### 3.5 建议优先级

P0。

这是产品的入口问题，优先级最高。

---

### 问题二：简历工具结果不可见，用户操作缺乏闭环

#### 3.1 用户视角表现

在“简历工具”页，用户点击：

- “评价已上传简历”
- “生成简历 PDF”

前端会提示：

- “已提交任务，请在对话区查看结果”

但实际很多情况下，聊天区看不到对应结果，尤其是简历评价。

用户会感知为：

- 按钮点了
- 任务似乎发出去了
- 但结果没有回来

#### 3.2 影响

- 用户无法确认任务是否真的执行成功。
- 用户无法拿到结构化评价结果。
- 产品会显得像“后台悄悄跑了点东西，但 UI 没接住”。

#### 3.3 根因

简历相关接口不是复用主 Agent，而是通过 `AgentFactory` 创建临时 Agent。

但当前 Web 侧的可观测链路只完整覆盖主 Agent：

- 主 Agent 的 channel 会转发流式输出到 WebSocket
- 主 Agent 会被注册到 `agentRegistry`
- 历史会话接口只读取注册 Agent 的消息

临时 Agent 没有完整接入上述链路，因此：

- 过程日志不完整
- 最终文本回复不进入聊天主界面
- 用户只能看到“任务已提交”的假完成状态

#### 3.4 修复目标

让“按钮提交任务”与“用户看见结果”形成完整闭环。

#### 方案 A：简历工具默认复用主 Agent

这是更简单、用户体验更稳定的方案。

实施方式：

1. `POST /api/resume/build` 改为调用主 Agent 提交消息。
2. `POST /api/resume/review` 改为调用主 Agent 提交消息。
3. 保持这些任务进入主会话队列，由 Web 原有聊天流直接展示结果。

优点：

- 不需要额外的多 Agent UI 设计。
- 结果天然进入聊天历史。
- 用户心智最简单：“我在工具页点了按钮，Agent 在主对话里继续处理。”

缺点：

- 主 Agent 忙时要排队。

但这个缺点对当前版本是可接受的。

#### 3.5 补充修复

无论选 A 还是 B，都建议补上：

1. 任务状态反馈：
   - 排队中
   - 执行中
   - 完成
   - 失败
2. 失败时在 UI 明确展示错误原因，而不是只打印服务端日志。
3. “请在聊天区查看结果”这类文案只在结果真的会进入聊天区时才保留。

#### 3.6 建议优先级

P0。

因为这是用户点击后最直观的“无响应”问题。

### 问题三：职位看板批量操作会覆盖 Agent 的新数据

#### 3.1 用户视角表现

用户在职位看板中进行：

- 批量标记已投递
- 批量标记失败
- 批量标记关注
- 删除

这些操作本应只是修改选中的那些职位。

但当前实现方式是：

- 前端拿当前浏览器里的整份 `jobs` 数据
- 本地修改几行
- 然后整表重新写回 `jobs.md`

如果这时 Agent 恰好又发现了几个新职位，或者后端刚更新了某些状态，用户下一次刷新时就可能发现：

- 刚看到的新职位消失了
- 某些状态被还原了
- 数据看起来“自己打架”

#### 3.2 影响

- 用户会认为数据不可靠。
- 批量操作越频繁，越容易和自动搜索流程互相覆盖。
- 看板和 Agent 之间不再是协作关系，而是竞争写入。

#### 3.3 根因

系统本来已经有 `upsert_job` 这种原子化写入工具，Agent 侧也被明确要求使用它。

但 Web 看板没有使用统一的职位写接口，而是绕过领域逻辑，直接通过 `/api/config/jobs` 全量重写文件。

锁只能保证“不会同时写崩文件”，不能保证“写入语义正确”。

也就是说：

- 现在解决了文件互斥
- 但没有解决用户语义上的并发冲突

#### 3.4 修复目标

把职位编辑从“整表覆盖”改为“领域级原子操作”。

#### 方案 A：新增职位专用 API，前端只提交变更意图

推荐方案。

新增接口示例：

- `POST /api/jobs/status`
  - 入参：`[{ url, status }]`
- `POST /api/jobs/delete`
  - 入参：`[{ url }]`
- `POST /api/jobs/upsert`
  - 单条或批量 upsert

后端处理原则：

1. 读取当前最新 `jobs.md`
2. 以 `url` 为主键合并变更
3. 保留其他未改动记录
4. 写回时仍使用锁

这样即使 Agent 刚新增了职位，也不会被前端老快照抹掉。

#### 3.5 补充修复

建议同时补上以下体验改进：

1. 批量删除前二次确认。
2. 保存成功后提示具体变更数量。
3. 如果后台数据已变化，提示“列表已更新，请确认后重试”。

#### 3.6 建议优先级

P0。

这直接影响用户对职位数据正确性的信任。

---

### 问题四：简历下载入口不反映真实文件状态

#### 3.1 用户视角表现

用户已经成功生成过一次简历，磁盘上也确实存在 `workspace/output/resume.pdf`，但只要刷新页面，UI 仍然显示：

- “尚未生成简历”

下载入口只有在本次页面会话内收到一次 `resume_ready` WebSocket 事件后才会显示。

这会让用户误以为：

- 简历文件丢了
- 需要重新生成

#### 3.2 影响

- 增加不必要的重复生成。
- 降低用户对文件产物持久性的预期。

#### 3.3 根因

前端没有在初始化时主动检查产物文件是否存在，而是完全依赖运行时事件驱动 UI。

#### 3.4 修复目标

让简历入口基于“真实状态”而不是“是否刚收到事件”。

#### 修复方案

新增一个初始化检查：

1. 新增接口：
   - `GET /api/resume/status`
   - 返回：
     - `exists`
     - `path`
     - `mtime`
2. 页面初始化时调用该接口。
3. 如果文件存在：
   - 直接显示“查看并下载 PDF”
4. 如果不存在：
   - 显示空状态
5. WebSocket 的 `resume_ready` 事件只负责实时更新，不再承担唯一数据来源角色。

#### 3.5 建议优先级

P1。

虽然不如前 3 个问题致命，但它会持续制造“文件状态不可信”的错觉。

---

## 4. 推荐实施顺序

### Phase 1：先修复主路径闭环

目标：让用户至少能顺畅完成“启动 -> 配置 -> 触发任务 -> 看见结果”。

建议顺序：

1. 修复首次使用流程
2. 修复简历任务结果不可见
3. 修复职位批量操作覆盖问题

### Phase 2：修复状态一致性

目标：让 UI 展示与真实文件状态一致。

建议顺序：

1. 增加简历状态接口
2. 优化按钮状态与失败提示
3. 补充初始化时的真实状态检查

---

## 5. 推荐落地改造清单

### 5.1 后端

- 调整 `validateEnv()`，支持 Web 引导模式或统一强校验策略。
- 为首次配置新增明确的状态接口或错误模型。
- 简历任务改为复用主 Agent，或为临时 Agent 补齐 Web channel 与注册机制。
- 新增职位专用变更 API，废弃 UI 对 `jobs.md` 的整表覆盖写回。
- 新增 `GET /api/resume/status`。

### 5.2 前端

- 首次运行 banner 改为真实引导，而不是仅提示填 markdown。
- 在未完成基础配置时，禁用聊天发送、简历生成、搜索相关交互。
- 简历工具页显示明确的任务状态和结果归属。
- 职位批量操作改为提交变更意图，而不是全量回写。
- 页面加载时主动检查简历文件状态。

### 5.3 文档

- README 必须与真实启动逻辑一致。
- 若采用“先配 config 再启动”，就不要再把 Web 说成首次配置入口。
- 若采用“可带缺省配置启动 Web”，就要把 UI 做成真正的引导模式。

---

## 6. 验收标准

修复完成后，至少应满足以下验收条件：

### 首次使用

- 新用户能清楚知道缺了什么配置。
- 新用户不会因为错误引导而卡死在启动前。

### 简历工具

- 点击“评价已上传简历”后，用户能在明确位置看到完整结果。
- 点击“生成简历 PDF”后，用户能稳定看到最终产物入口。

### 职位看板

- 用户批量修改职位状态时，不会把后台新增职位覆盖掉。
- 批量删除不会无提示地造成误删。

### 状态一致性

- 刷新页面后，若 `resume.pdf` 已存在，下载入口仍然存在。
- UI 空状态与真实文件状态一致。
