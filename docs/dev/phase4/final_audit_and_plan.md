## 一、 综合审查结果 (Audit Results)

### 1. 总体评价
**结论**: ✅ **核心链路已闭环，架构稳健性达标。**
系统成功整合了双 Agent 协同、TUI 实时监控、HITL 挂起超时机制以及 MCP 浏览器自动化。目前已具备从实验性工具向生产力工具转化的基础。

### 2. 核心发现清单

| 类别 | 发现项 | 优先级 | 影响 | 建议措施 |
| :--- | :--- | :--- | :--- | :--- |
| **鲁棒性** | `validateWorkspace` 已实现但未在入口调用 | **P0** | 配置错误时带病启动，反馈滞后 | 在 `index.ts/cron.ts` 强制启用预检 |
| **一致性** | `upsert_job` 锁持有者标识硬编码为 `'system'` | **P1** | 无法准确区分是哪个 Agent 在操作文件 | 改为使用 `context.agentName` |
| **集成** | MCP 工具返回 JSON 数组结构，消耗额外 Token | **P1** | 浪费 LLM 上下文空间，增加响应成本 | 优化 `mcp.ts` 逻辑，仅提取并返回纯文本 |
| **可靠性** | 缺乏工具调用和投递的自动重试机制 | **P1** | 偶发网络抖动会导致任务直接标记失败 | 增加 3 次自动重试逻辑 |
| **体验** | TUI 在 Agent 运行时缺乏忙碌状态提示 | **P2** | 用户无法直观判断 Agent 是否正在思考 | 增加视觉 Loading 状态（如 Label 变化） |
| **配置** | `maxIterations` 默认 50 次在复杂场景可能偏低 | **P2** | 深度投递流程可能中途截断 | 将默认值上调至 100 |

## 二、 后续开发路线图 (Development Roadmap)

### 1. Phase 4.2: 最终润色 (Final Polish) - 立即执行
**目标**: 修复审计发现的 P0/P1 级瑕疵，完成 Phase 4 的完美结项。

- [ ] **[P0] 启用启动预检**: 在 `index.ts` 和 `cron.ts` 中补全 `validateWorkspace` 调用。
- [ ] **[P1] 动态锁持有者**: 重构 `upsertJob` 逻辑，从上下文动态获取 `agentName`。
- [ ] **[P1] MCP 数据精简**: 修改 `mcp.ts`，对 `callTool` 的返回结果进行文本提取。
- [ ] **[P1] 增加重试指引**: 在 `jobclaw-skills.md` 的投递 SOP 中显式增加重试逻辑说明。

### 2. Phase 5: Web UI & Monitoring - 当前重点
**目标**: 提供可视化、低门槛的监控与配置面板。

- [ ] **Web 基础设施**: 搭建基于 `Hono` 的轻量级 Web 服务器。
- [ ] **数据可视化**: 提供 `jobs.md` 的实时统计 API 和 Web 表格展示。
- [ ] **交互式配置**: 在网页端直接编辑 `targets.md` 和 `userinfo.md`。
- [ ] **操作日志**: Web 端的 Agent 活动流同步显示。

### 3. Phase 6: 高级生产特性 - 远期规划
**目标**: 提升系统在极端情况下的稳定性与大规模运行效率。

- [ ] **TUI 渲染性能**: 引入内容哈希校验，减少 `jobs.md` 无效解析。
- [ ] **自动化重试框架**: 在 `BaseAgent` 层面实现工具调用的指数退避重试。
- [ ] **Session 智能管理**: 定期清理冗余的消息历史，保持 Session 紧凑。
- [ ] **通道限流**: 针对邮件通知增加发送频率保护逻辑。
