# JobClaw 开发计划

> 最近更新：2026-03-27  
> 说明：本文件只保留“当前仍未完成”或“明显需要继续演进”的事项；已完成的历史改造计划不再单独保留。

## 1. 当前实现基线

以下能力已经落地，不再作为待做项描述：

- Web 默认启动模式
- `workspace/config.json` 扁平配置
- `AgentFactory + run_agent` 子任务模型
- `request` 工具与人工干预事件链路
- PDF 简历上传与 `read_pdf`
- 简历评价与模拟面试入口
- Web 看板、实时日志、配置编辑、简历生成

## 2. 当前仍需推进的方向

### 2.1 稳定性

- 工具调用自动重试：当前工具失败会直接返回，需要为可重试错误补指数退避。
- MCP 不可用时的降级体验：现在主要靠 prompt 和日志提醒，还缺少更明确的用户侧状态与指导。
- 长时间运行 session 管理：虽然已有 `ContextCompressor`，但 `session.json` 仍可能持续膨胀。

### 2.2 交互与展示

- Web 与 TUI 的事件语义仍有历史兼容层，后续可以继续收敛，减少双轨逻辑。
- `agent:stream`、`agent:log`、`agent:tool` 的前端展示仍可进一步统一，避免同一任务在不同面板出现重复语义。
- TUI 已不是默认入口，但仍保留实现；后续要么补齐体验，要么继续明确降级定位。

### 2.3 业务链路

- 职位搜索仍高度依赖 MCP/Playwright，缺少在浏览器不可用时的替代路径。
- 投递链路主要依赖 skill 和浏览器工具，缺少更明确的阶段状态建模与回放能力。
- `jobs.md` 仍是 Markdown 表驱动，后续如果字段继续增长，需要重新评估结构化存储边界。

### 2.4 测试与文档

- 文档需要继续保持和代码同步，尤其是 `README.md` 与 `docs/` 的行为说明。
- 对 Web API、事件协议和子 Agent 调度路径仍应持续补回归测试。
- 若继续保留 TUI，应补对应的回归测试和维护标准；否则应在文档中进一步弱化其主路径地位。

## 3. 近期建议顺序

1. 先补稳定性：工具重试、session 管理、MCP 降级提示。
2. 再收敛展示层：统一流式消息与工具日志语义。
3. 最后评估业务数据模型：是否继续坚持纯 Markdown 驱动，还是为职位与投递状态引入更强结构化层。

## 4. 文档维护规则

- 功能一旦完成，应直接删掉对应的“计划文档”，不要长期保留“已完成计划”。
- 架构变更后，至少同步更新 `docs/SPEC.md` 与 `docs/agent-design.md`。
- `*-review.md` 只保留评审语义，不改写成功能规格说明书。
