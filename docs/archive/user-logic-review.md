# JobClaw 用户逻辑审查（历史归档）

## 1. 文档定位

本文件保留的是一次关键用户视角审查的历史结论。

其中提出的 3 个核心问题曾经非常重要，但其中大部分已经在代码中得到修正或部分修正，因此这里不再保留原始长篇整改方案，避免和当前实现混淆。

## 2. 当时识别出的 3 个核心问题

### 2.1 首次配置与服务启动冲突

历史问题：

- 用户被引导通过 Web 完成初始化
- 但服务启动前会因缺少模型配置而直接失败

当前状态：

- 该问题已基本修复
- Web 模式允许缺少基础模型配置启动
- 当前系统会进入设置向导状态，而不是直接启动失败

仍然值得保留的启发：

- 首屏必须明确告诉用户“还缺什么”和“下一步做什么”

### 2.2 简历工具结果不可见

历史问题：

- 简历任务通过临时 Agent 执行
- 结果无法稳定回到用户当前可见的主界面

当前状态：

- 该问题已部分修复
- `/api/resume/build` 和 `/api/resume/review` 在主 Agent 存在时会优先提交给主 Agent
- 但前端层面的统一任务状态与结果落点仍未完全收口

仍然值得保留的启发：

- “任务已提交”不等于“用户获得闭环”

### 2.3 职位整表覆盖写回

历史问题：

- 前端通过整表重写 `jobs.md`
- 容易与 Agent 的增量写入产生冲突

当前状态：

- 该问题已部分修复
- 当前已有 `POST /api/jobs/status` 和 `POST /api/jobs/delete`
- 但 `/api/config/jobs.md` 仍允许整表覆盖，因此领域级写入和通用文件编辑仍并存

仍然值得保留的启发：

- 面向用户的高频数据操作，应优先走领域接口而不是通用文件编辑接口

## 3. 这份历史审查留下的长期价值

这份审查虽然已经不是当前 backlog，但仍然留下了 3 条长期成立的产品判断：

- 用户首先关心的是能不能顺利开始
- 用户其次关心的是点完之后有没有结果
- 用户最终关心的是数据能不能信任

- `docs/pm/product-strategy-and-operating-plan.md`
