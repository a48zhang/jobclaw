# JobClaw 前端问题审查与修复计划

> 审查日期：2026-03-27  
> 审查范围：`public/index.html`、`public/css/style.css`、`public/js/*.js`  
> 审查目标：从真实用户使用 Web 控制台的视角，识别当前前端在可用性、可访问性、响应式、视觉一致性和任务闭环上的不足，并给出分阶段修复计划。

## 1. 总体结论

当前前端已经具备可演示的产品骨架，核心模块也比较完整：

- 智能助理
- 职位看板
- 工作区配置
- 简历工具

但整体体验仍停留在“功能可点通”的阶段，还没有进入“高频使用也顺手”的状态。当前最明显的问题集中在 6 个方面：

1. 页面骨架依赖固定视口高度，移动端和小窗口下容易挤压、溢出、双滚动。
2. 可访问性基础不完整，键盘操作、焦点状态、模态框语义和表格交互都有缺口。
3. 视觉系统不统一，终端感、工具感和产品感混杂，信息层级不够稳。
4. 用户反馈机制不一致，任务发起、排队、执行和失败没有统一的表达方式。
5. 职位看板缺少高频处理工具，信息组织仍偏“展示”，不够“操作台”。
6. 动效与资源加载策略还不够稳，对长期使用和弱网环境不友好。

## 2. 建议的前端设计基线

结合当前产品定位和 `ui-ux-pro-max` 的设计系统建议，JobClaw 更适合采用一套更稳定、更偏工作台的设计基线：

- 产品类型：求职助手 / SaaS 工作台
- 推荐风格：Flat Design
- 视觉关键词：简洁、清晰、信息优先、轻量动效、低装饰
- 推荐主色：Teal 系
- 推荐动作色：Orange 系
- 字体方向：`Plus Jakarta Sans` 作为主字体，等宽字体只用于代码和日志
- 动效原则：只保留必要状态过渡，控制在 150-200ms，并支持 `prefers-reduced-motion`

当前实现与这套基线相比，最主要的问题不是“缺少设计”，而是“设计语言没有收敛”。

## 3. 详细问题

### 3.1 响应式骨架不稳，依赖固定视口高度

#### 表现

- 聊天区高度写死为 `calc(100vh - 180px)`。
- 职位区和配置区都依赖 `h-[calc(100vh-140px)]`。
- 顶部导航和批量操作区在窄屏下没有针对性重排策略。
- CSS 的移动端兜底只处理了聊天容器和消息宽度，没有处理整体页面骨架。

#### 代码位置

- `public/css/style.css`
- `public/index.html`

相关实现：

- `public/css/style.css:299`
- `public/index.html:80`
- `public/index.html:125`
- `public/index.html:16`
- `public/index.html:81`

#### 影响

- 小屏和低高度窗口中，内容区会被顶部区和操作区挤压。
- 容易出现容器内滚动和页面滚动并存，阅读与操作都不流畅。
- 职位页的批量按钮在窄屏下可点击性下降。
- 后续继续加功能时，布局会越来越脆弱。

#### 修复方向

- 改成真正的弹性布局，不再把主要模块绑死在 `100vh` 计算上。
- 页面改为“头部固定信息 + 内容区自适应 + 模块内部再滚动”的结构。
- 为 `375px / 768px / 1024px / 1440px` 制定明确断点策略。
- 职位页顶部操作区改为多行布局或折叠式操作栏。

### 3.2 可访问性基础不完整

#### 表现

- 排序表头通过可点击的 `<th>` 实现，不是按钮，也没有 `aria-sort`。
- 导航按钮、日志按钮、批量按钮没有统一的焦点样式。
- 输入框大量使用 `outline-none`，但没有一套完整的 `focus-visible` 规范。
- 模态框缺少 `role="dialog"`、`aria-modal="true"`、焦点陷阱和关闭后的焦点恢复。
- 页面中大量依赖颜色表达状态，没有统一的语义补充。

#### 代码位置

- `public/index.html`
- `public/js/jobs.js`
- `public/js/modal.js`

相关实现：

- `public/index.html:99`
- `public/js/jobs.js:161`
- `public/index.html:221`
- `public/js/modal.js:60`
- `public/index.html:56`
- `public/index.html:179`
- `public/index.html:232`

#### 影响

- 键盘用户难以稳定操作排序、模态和主导航。
- 屏幕阅读器很难理解交互语义。
- 焦点丢失会让高频用户产生“点了但没反应”的感知。
- 后续如果增加更多弹窗和快捷操作，可访问性问题会继续扩大。

#### 修复方向

- 把可交互表头改为 `<button>`，并同步维护 `aria-sort`。
- 补一套全局 `focus-visible` 方案，不再只改单个输入框边框。
- 模态框补齐语义属性、焦点陷阱、Esc 关闭和焦点恢复。
- 对状态标签、连接状态、错误状态增加文本层和无障碍语义层。

### 3.3 视觉系统不统一，产品感不足

#### 表现

- 页面根部使用 `font-mono`，使整个产品带有过强的终端感。
- 标题、导航、按钮、成功态、警告态大量使用 emoji。
- 主色在蓝、绿、黄、紫、红之间切换，没有一套稳定的颜色职责划分。
- 背景、气泡、卡片和按钮同时使用渐变、阴影、玻璃感和终端感，风格混杂。

#### 代码位置

- `public/index.html`
- `public/css/style.css`

相关实现：

- `public/index.html:14`
- `public/index.html:17`
- `public/index.html:19`
- `public/index.html:203`
- `public/index.html:224`
- `public/css/style.css:3`
- `public/css/style.css:200`

#### 影响

- 首页第一眼像“实验工具”而不是“可靠工作台”。
- 用户很难快速分辨“品牌色”“主操作色”“危险操作色”。
- 信息层级不稳定，注意力经常被装饰性元素带走。
- 视觉风格越往后扩展，越容易失控。

#### 修复方向

- 统一品牌语气，减少 emoji，优先使用 SVG 图标。
- 页面主字体改为适合 SaaS 工作台的无衬线字体。
- 颜色角色收敛为：主色、辅助色、成功、警告、危险、边框、中性文本。
- 聊天气泡、面板、日志区采用同一套圆角、边框、阴影和层级规则。

### 3.4 任务反馈不一致，用户预期难建立

#### 表现

- 有的操作把结果写进聊天区。
- 有的操作写在日志区。
- 有的操作直接 `alert()` 或 `confirm()`。
- 简历工具会在提交后切回聊天页，打断当前上下文。
- 聊天非排队模式下会直接清空历史内容。

#### 代码位置

- `public/js/chat.js`
- `public/js/jobs.js`
- `public/js/resume.js`

相关实现：

- `public/js/chat.js:150`
- `public/js/chat.js:155`
- `public/js/jobs.js:144`
- `public/js/resume.js:77`
- `public/js/resume.js:124`
- `public/js/resume.js:134`

#### 影响

- 用户很难判断任务到底是否发起成功。
- 用户需要自己猜测“结果应该去哪里看”。
- 当前页被强制切走，会打断操作链路。
- 历史内容被清空会降低会话可信度。

#### 修复方向

- 统一前端反馈层，明确区分：
  - 全局任务状态
  - 当前页面操作状态
  - 系统错误
  - 持久日志
- 禁止在主要用户路径中继续使用 `alert()` / `confirm()`。
- 简历任务提交后保留当前页，只提供“查看任务进度”的跳转入口。
- 聊天区保留历史，不再因为某种返回模式直接整体清空。

### 3.5 职位看板更像展示表，而不是操作台

#### 表现

- 有排序和批量操作，但没有筛选、搜索、状态切换、快速预览。
- 高价值信息都挤在同一张表里，没有行级操作层次。
- 批量按钮过多，集中放在同一层，窄屏下可读性和可点击性都一般。
- 图表区有统计，但不能反过来驱动筛选和处理。

#### 代码位置

- `public/index.html`
- `public/js/jobs.js`

相关实现：

- `public/index.html:81`

- `public/js/jobs.js:30`

#### 影响

- 用户要做“筛选待处理职位 -> 快速查看 -> 批量更新状态”时路径不顺。
- 图表只是装饰性统计，还没有形成真正的任务分发入口。

#### 修复方向

- 增加状态筛选、关键字搜索和快捷过滤。
- 提供行级详情预览或侧边抽屉。
- 将图表点击映射到筛选动作。
- 批量操作改为有主次的工具栏，而不是一排平权按钮。

### 3.6 动效和资源策略不够稳

- 未适配 `prefers-reduced-motion`。
- 核心依赖通过 CDN 直接加载。
- 当前前端在网络较差或 CDN 抖动时存在不稳定性风险。

#### 代码位置

- `public/index.html`

- `public/css/style.css:108`
- `public/index.html:7`
- `public/index.html:8`

#### 影响

- 动效对长期使用者和敏感用户不友好。
- 弱网下页面首屏和交互稳定性会下降。
- 线上环境的确定性较弱。

#### 修复方向

- 为动画统一加上 `prefers-reduced-motion` 降级。
- 只保留必要的状态动画，减少持续型装饰动画。
- 评估将关键依赖本地化或纳入构建流程。

## 4. 修复计划

### 阶段 0：先收敛设计系统和交互原则

#### 目标

在动代码前，先明确一套不会反复返工的前端规范。

#### 交付物

- 前端视觉变量表
- 组件状态规范
- 页面断点策略
- 反馈层级规范

#### 计划内容

1. 收敛字体方案：
   - 主字体改为无衬线字体
   - 等宽字体仅用于日志、代码、路径和少量技术信息
2. 收敛颜色职责：
   - 主品牌色
   - 主操作色
   - 成功 / 警告 / 危险
   - 中性背景 / 边框 / 文本
3. 收敛组件样式：
   - 按钮尺寸
   - 输入框焦点
   - 卡片圆角
   - 阴影和边框层级
4. 明确反馈规范：
   - 页面内状态提示
   - 全局 toast
   - 日志区
   - 长任务状态

#### 优先级

P0。

如果这一步不做，后续修复会继续产生风格漂移。

### 阶段 1：修正页面骨架、响应式和可访问性

#### 目标

先把“能稳定使用”这件事做好。

#### 计划内容

1. 重构主布局
   - 去掉核心区域对 `100vh` 的硬编码依赖
   - 统一模块最小高度与内部滚动边界
   - 为窄屏重排头部和操作栏
2. 补齐焦点体系
   - 全局 `focus-visible`
   - 键盘导航顺序检查
   - 可交互元素统一可见焦点
3. 改造表头排序语义
   - `<th>` 内包裹按钮
   - 同步 `aria-sort`
   - 提供更清晰的视觉状态
4. 改造模态框
   - 加 `role="dialog"` 和 `aria-modal`
   - 打开时聚焦首个可交互元素
   - 关闭后恢复焦点
   - 增加焦点陷阱
5. 处理动效降级
   - 为消息动画和流式状态增加 `prefers-reduced-motion`

#### 验收标准

- `375px` 宽度下主页面无横向滚动
- 键盘可完整操作主导航、聊天输入、职位筛选和模态框
- 模态框打开关闭后焦点行为正确
- 页面无明显双滚动冲突

#### 优先级

P0。

### 阶段 2：统一状态反馈和长任务闭环

#### 目标

让用户始终知道“发生了什么”“现在在哪看结果”。

#### 计划内容

1. 引入统一的页面消息机制
   - 替代 `alert()`
   - 替代原生 `confirm()`
   - 区分成功、失败、警告、信息
2. 统一任务状态表达
   - 已提交
   - 排队中
   - 执行中
   - 完成
   - 失败
3. 优化简历工具链路
   - 提交后保留在当前页
   - 展示任务状态和结果入口
   - 不再强制切换到聊天页
4. 优化聊天区返回逻辑
   - 保留会话历史
   - 将非流式消息也纳入统一消息模型

#### 验收标准

- 任一任务提交后，用户都能在当前页看到状态变化
- 任一失败都能在 UI 中直接看到原因
- 聊天历史不会因接口返回分支被整块清空

#### 优先级

P0。

### 阶段 3：重做职位看板为“操作台”

#### 目标

让职位页面从“看数据”升级为“高效处理数据”。

#### 计划内容

1. 增加筛选工具
   - 按状态筛选
   - 按公司 / 职位关键字搜索
   - 最近更新时间排序与过滤
2. 增加行级预览
   - 抽屉或侧栏展示职位详情
   - 直接显示链接、状态、更新时间和备注位
3. 重构批量操作条
   - 区分主操作和危险操作
   - 选中状态下再展开完整操作
4. 让统计图可交互
   - 点击图例或扇区后联动筛选结果

#### 验收标准

- 用户可以在 3 步内完成“筛选待处理职位 -> 查看详情 -> 批量更新状态”
- 批量操作区在窄屏下仍可用
- 图表不再只是展示，而是可作为筛选入口

#### 优先级

P1。

### 阶段 4：视觉统一和资源稳定性优化

#### 目标

把前端从“能用”提升到“可长期使用且有产品完成度”。

#### 计划内容

1. 清理 emoji 型图标，替换为统一 SVG 图标集
2. 收敛气泡、卡片、按钮、标签的视觉样式
3. 将主字体和图标体系稳定下来
4. 评估把关键前端依赖从 CDN 改为本地依赖或构建产物

#### 验收标准

- 页面不再依赖 emoji 传达主要语义
- 同一类组件在不同页面的样式一致
- 弱网下页面初始化稳定性更高

#### 优先级

P1。

## 5. 推荐实施顺序

建议按下面顺序推进，而不是分页面零散修补：

1. 先做阶段 0，固定设计系统和反馈规范。
2. 再做阶段 1，解决布局、响应式和可访问性问题。
3. 然后做阶段 2，统一任务状态和长任务反馈。
4. 再做阶段 3，重构职位看板的操作体验。
5. 最后做阶段 4，统一视觉细节和资源稳定性。

## 6. 立即可执行的首批任务

如果要开始落地，建议先拆下面这些任务：

1. 建立全局 CSS 变量和组件状态变量，替换零散颜色与阴影。
2. 去掉主区域固定高度，重构页面骨架为弹性布局。
3. 为按钮、输入框、导航、表头补齐统一焦点态。
4. 将排序表头改造成按钮语义。
5. 把原生 `alert()` / `confirm()` 替换成统一的前端反馈组件。
6. 改造简历页提交逻辑，取消强制跳转聊天页。
7. 为职位页增加状态筛选和关键字搜索。

## 7. 结论

当前前端的主要问题并不是“没有页面”，而是“页面已经有了，但缺少一套统一的产品级前端规则”。  
接下来的重点不应该是继续堆功能，而应该是先把基础交互和信息结构稳定下来。只有这样，后续新增功能才不会继续放大现有体验问题。
