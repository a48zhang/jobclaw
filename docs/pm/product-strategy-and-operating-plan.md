# JobClaw 现在与未来产品计划

> 版本：0.2  
> 日期：2026-03-27  
> 配套文档：`docs/pm/product-direction-and-requirements.md`

## 1. 文档目的

本文件只回答两件事：

- 现在做什么
- 将来做什么

不展开 PM 方法论，不展开流程管理，重点是把产品知识库收敛成可执行的方向。

## 2. 产品一句话

JobClaw 要做的，不是一个“会点网页的 Agent 集合”，而是一个可信、可解释、可持续协作的求职系统。

当前这一阶段的产品入口应明确为：

- 用户先与 Agent 对话。
- Agent 在对话中维护 `targets.md` / `userinfo.md` 等工作区上下文。
- 只有当缺失信息已经影响搜索范围、简历策略或自动化安全时，Agent 才追问用户。

## 3. 现在做什么

当前阶段的核心目标只有 3 个：

- 让新用户第一次使用时不迷路
- 让所有主要任务都有明确闭环
- 让高频求职处理效率真正提升

### 3.1 现在的主线一：可信入口

现在要做：

- 聊天优先的首次成功引导
- 缺失配置和能力状态展示
- Agent 自动补齐上下文时的可见反馈
- 未完成基础配置时的功能禁用与解释
- 明确的“下一步做什么”提示

要解决的问题：

- 用户不知道第一步该做什么
- 用户不知道哪些功能当前可用
- 用户把“Agent 可以代为补齐的上下文”误解为“必须先手工填完的前置表单”

完成标志：

- 用户进入首页后能快速知道下一步动作
- 用户完成基础配置后直接进入对话，而不是先被驱赶到资料编辑页
- 不需要翻日志就能理解当前状态

### 3.2 现在的主线二：任务闭环

现在要做：

- 统一任务状态模型
- 统一结果落点
- 统一失败原因与重试入口
- 保留当前页面上下文，不强制切页

要解决的问题：

- 用户点了按钮，不知道系统是否真的开始执行
- 用户不知道结果该去哪里找
- 简历、搜索、投递、人工干预的反馈不一致
- Agent 自动更新上下文时，用户不知道它改了什么、为什么改

完成标志：

- 任意长任务都能看到 `已提交 / 排队中 / 执行中 / 完成 / 失败 / 需人工介入`
- 用户不再依赖日志区判断任务是否成功

### 3.3 现在的主线三：职位处理台

现在要做：

- 状态筛选
- 关键词搜索
- 最近更新时间排序
- 行级详情预览
- 更合理的批量操作工具栏
- 图表与列表联动

要解决的问题：

- 看板更像展示页，不像处理台
- 职位一多，用户处理效率迅速下降
- 图表只是统计，不是操作入口

完成标志：

- 用户可以高频处理大量职位
- “筛选 -> 查看 -> 批量推进”成为顺手路径

### 3.4 现在的主线四：数据可信与降级可见

现在要做：

- 职位改动从整表覆盖改为原子操作
- 展示最近更新时间和变更来源
- 在 UI 中明确提示 MCP、浏览器、Typst、模型配置状态
- 为不可用能力提供替代说明

要解决的问题：

- 用户担心 Agent 和手工操作互相覆盖
- 用户不知道某项能力为什么不能用

完成标志：

- 用户能信任职位数据不会莫名丢失
- 用户能明确知道当前系统缺什么、受限什么

### 3.5 现在暂时不优先做

当前不应优先投入的方向：

- 只对演示效果有帮助的新花样 UI
- 依赖不稳定浏览器能力但没有降级方案的新自动化
- 与主求职路径关系不大的边缘能力
- 复杂但暂时没有用户闭环的多 Agent 展示层

## 4. 将来做什么

当“可信入口 + 任务闭环 + 职位处理台”站住以后，再往下做这 4 层能力。

### 4.1 将来做：投递与跟进管理

未来要做：

- 投递状态时间线
- 跟进提醒与待办
- 拒信记录
- 公司和岗位维度的进度视图
- 可回放的投递执行轨迹

目标：

- 让 JobClaw 从“找到职位”走向“推进求职过程”

### 4.2 将来做：简历工作流深化

未来要做：

- 岗位定制版简历
- 简历版本管理
- JD 差距分析
- 产物历史与效果关联

目标：

- 让简历模块从“几个工具按钮”变成持续优化的工作流

### 4.3 将来做：求职策略层

未来要做：

- 目标岗位篮子
- 城市和薪资偏好
- 公司优先级
- 投递节奏管理
- 机会评分和推荐顺序

目标：

- 让系统不只是帮用户执行动作，也帮用户做更好的求职判断

### 4.4 将来做：学习与成长闭环

未来要做：

- 面试模拟
- 失败归因
- 简历与岗位命中率分析
- 基于历史记录的改进建议

目标：

- 让 JobClaw 从执行系统走向学习系统

## 5. 分阶段路线

### 阶段一：先站住现在

范围：

- 可信入口
- 任务闭环
- 职位处理台基础版
- 数据可信与降级可见

阶段目标：

- 用户第一次使用不迷路
- 用户主要任务不失联
- 用户开始信任系统数据

### 阶段二：再扩展主路径

范围：

- 简历工作流深化
- 聊天区重构
- 职位处理台增强版
- 视觉与交互统一

阶段目标：

- 用户可以每天稳定高频使用

### 阶段三：最后做长期护城河

范围：

- 投递与跟进管理
- 求职策略层
- 学习与成长闭环

阶段目标：

- JobClaw 从工作台升级为持续协作的求职系统

## 6. 未来持续关注的问题

这些问题不是现在就要全部做完，但应持续观察：

- 聊天入口是否已经足够承载上下文维护与任务推进，而不必把用户赶去表单页
- 哪些任务用户愿意完全放手自动化
- 职位优先级判断最该依据什么
- 用户更看重简历生成、简历评价还是岗位定制
- 哪些执行轨迹应该展示，哪些只会制造噪音

## 7. 一句话结论

现在先把信任、闭环和效率做扎实；将来再把投递、策略和学习做成护城河。
