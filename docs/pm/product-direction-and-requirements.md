# JobClaw 产品问题与需求方向

> 版本：0.1  
> 日期：2026-03-27  
> 角色视角：产品经理  
> 参考资料：`docs/SPEC.md`、`docs/archive/user-logic-review.md`、`docs/archive/frontend-ui-review.md`、`docs/archive/frontend-ui-task-list.md`、`docs/dev/plan.md`、`docs/agent-first-architecture.md`

## 1. 文档目的

本文件不讨论底层实现细节，而是从用户视角回答 4 个问题：

- JobClaw 当前到底是什么产品。
- 用户现在最容易在哪些地方失去信任。
- 团队下一阶段最值得做哪些需求。
- 这些需求完成后，产品应达到什么状态。

## 2. 当前产品阶段判断

当前的 JobClaw 更像“求职 Agent 工作台”，还不是“可放心托管的自动求职助手”。

已经具备的产品骨架：

- Web 控制台是默认入口。
- 有主对话、职位看板、配置页、简历工具和日志区。
- 支持职位搜索、简历生成、简历评价、人工干预和子任务调度。

当前最核心的产品短板不在“功能数量”，而在以下 3 个用户感受还没有建立起来：

- 用户知道现在该做什么。
- 用户知道系统刚刚做了什么。
- 用户相信系统不会把数据弄乱。

如果这 3 点不能先建立，继续扩能力只会放大复杂度，不会放大产品价值。

## 3. 核心用户与主任务

### 3.1 新用户

主任务：

- 完成首次配置。
- 补齐求职资料。
- 发起第一次搜索或简历任务。

核心诉求：

- 快速上手。
- 不踩配置坑。
- 知道每一步的前置条件。

### 3.2 高频求职用户

主任务：

- 持续搜索职位。
- 筛选和管理职位状态。
- 针对岗位调整简历。

核心诉求：

- 高效率处理大量职位。
- 任务状态稳定可追踪。
- 数据可信且不互相覆盖。

### 3.3 重度自动化用户

主任务：

- 放手让 Agent 搜索和投递。
- 只在关键节点介入。
- 复盘 Agent 的行为和结果。

核心诉求：

- 明确的能力边界。
- 可靠的降级策略。
- 可解释的自动化。

## 4. 当前核心产品问题

### 4.1 首次成功路径不够强

当前问题：

- Web 可以启动，但用户不一定知道先配什么、后配什么。
- 运行配置、业务资料和功能可用性之间的关系不够直观。
- 用户很容易把“未完成配置”理解为“产品坏了”。

用户感受：

- 我看到了很多入口，但不知道第一步该点哪里。
- 我不知道哪些能力已经能用，哪些还不能用。

需要的产品动作：

- 建立首屏 onboarding。
- 明确显示缺失配置项和功能解锁关系。
- 让“首次成功”成为一个可完成的路径，而不是文档假设。

### 4.2 任务反馈不统一，闭环不足

当前问题：

- 用户动作的结果有时出现在聊天区，有时出现在日志区，有时只看到提交成功提示。
- 长任务没有统一状态模型。
- 简历、搜索、投递、人工干预这些动作还没有形成一致的反馈语言。

用户感受：

- 我点了按钮，但不知道系统是否真的开始工作。
- 我不知道结果应该去哪里找。

需要的产品动作：

- 建统一任务状态。
- 建统一结果落点。
- 建统一失败表达和重试入口。

### 4.3 职位看板还是展示页，不是处理台

当前问题：

- 看板有表格和批量操作，但缺筛选、搜索、详情预览和强任务流。
- 高价值动作还不能形成顺手的高频处理路径。
- 图表更多是统计展示，还没有成为操作入口。

用户感受：

- 我能看到职位，但不方便快速处理大量职位。
- 数据多起来以后，操作成本会上升得很快。

需要的产品动作：

- 强化筛选和搜索。
- 提供行级详情与快捷处理。
- 让图表和列表形成联动。

### 4.4 自动化承诺与运行现实之间还有落差

当前问题：

- 浏览器能力、MCP、Typst、受限子 Agent 等能力仍在继续收口。
- 用户看到的是“有能力入口”，但不一定知道当前环境是否真正可用。
- 降级状态更多体现在日志和技术说明里，而不是用户可理解的界面提示。

用户感受：

- 这个系统说自己能做很多事，但我不确定它现在到底能做哪些。

需要的产品动作：

- 明确能力状态。
- 明确降级原因。
- 明确替代路径和恢复方法。

### 4.5 产品心智还没有完全收敛

当前问题：

- 聊天助手、工作台、自动执行器、日志控制台四种形态并存。
- 页面和反馈语言还没有统一的产品叙事。

用户感受：

- 这是一个聊天产品、一个后台工具，还是一个自动化代理平台？

需要的产品动作：

- 收敛主叙事。
- 明确主界面层级。
- 让每个区域只承担一种核心职责。

## 5. 需求优先级

## 5.1 P0：先建立可信入口与任务闭环

### 需求 P0-1：首次配置引导

目标：

- 让用户能在 Web 内完成首次成功。

核心需求：

- 首页展示 onboarding 卡片。
- 明确展示缺失的运行配置项。
- 明确展示 `targets.md` 和 `userinfo.md` 的完成状态。
- 未完成基础配置时，禁用不该使用的功能并解释原因。

用户故事：

- 作为第一次使用 JobClaw 的用户，我希望页面明确告诉我还缺什么，这样我能在第一次访问时完成基本配置并开始使用。

验收标准：

- 用户进入首页后 30 秒内能知道下一步动作。
- 未配置完成时不会误触不可用能力。
- 配置完成后页面明确显示“可以开始搜索”。

### 需求 P0-2：统一任务状态模型

目标：

- 让所有长任务都具备一致的状态表达。

核心需求：

- 建立 `已提交 / 排队中 / 执行中 / 完成 / 失败 / 需人工介入` 六态模型。
- 聊天、简历、搜索、投递都复用同一套任务状态。
- 失败时给出明确原因和可执行下一步。

用户故事：

- 作为用户，我希望任何任务都有清晰状态，这样我不用猜系统是否真的在执行。

验收标准：

- 任意长任务都能在 UI 中看到状态流转。
- 用户不需要依赖日志区判断任务是否成功。
- 失败任务都能看到失败原因。

### 需求 P0-3：统一结果落点与任务中心

目标：

- 让用户始终知道结果去哪里找。

核心需求：

- 建立统一任务中心或统一结果面板。
- 页面内任务保留当前上下文，不强制切页。
- 提供结果入口、最近产物入口和最近失败记录。

用户故事：

- 作为用户，我希望我从任何页面发起任务，都能在固定位置看到结果，而不是自己猜结果落在哪个模块。

验收标准：

- 简历任务、搜索任务、投递任务的结果都可以从统一入口查看。
- 页面不再因为任务发起而强制跳转到聊天页。
- 结果位置和任务状态位置保持一致。

### 需求 P0-4：职位数据改为领域级原子操作

目标：

- 让用户相信职位数据不会被并发操作覆盖。

核心需求：

- 前端不再通过整表重写方式更新职位。
- 提供职位级状态更新、删除和 upsert 的专用接口。
- 对用户清晰表达最近更新来源和更新时间。

用户故事：

- 作为用户，我希望我手动修改职位状态时，不会把 Agent 刚发现的新职位覆盖掉。

验收标准：

- 批量操作不会覆盖无关职位。
- Agent 自动写入与用户手动写入可以共存。
- 职位更新时间与变更来源可追踪。

### 需求 P0-5：能力降级与环境状态面板

目标：

- 让用户知道当前环境哪些能力可用，哪些不可用。

核心需求：

- 首页或状态区展示模型配置状态、浏览器能力状态、Typst 状态。
- 对不可用能力给出恢复建议。
- 对降级模式给出替代路径。

用户故事：

- 作为用户，我希望系统明确告诉我当前哪些能力不可用，以及我要怎么恢复，而不是只在日志里看到技术提示。

验收标准：

- 关键能力的可用性都可以在 UI 中看到。
- 用户不需要打开日志也能理解当前限制。
- 降级状态不会误导用户点击无效入口。

## 5.2 P1：提升高频求职效率

### 需求 P1-1：职位看板升级为处理台

核心需求：

- 增加状态筛选、关键词搜索、最近更新时间排序。
- 增加行级详情预览。
- 批量操作按主次重构。
- 图表点击可反向触发筛选。

价值：

- 支持高频职位处理。
- 提升每日工作效率。

### 需求 P1-2：简历工作流产品化

核心需求：

- 把上传简历、评价简历、生成 PDF、下载产物拆成清晰步骤。
- 展示最近任务状态、最近产物、失败原因和重试入口。
- 建立岗位定制版简历的产物记录和版本意识。

价值：

- 把“几个按钮”变成“完整工作流”。
- 提升简历模块的可理解性和复用率。

### 需求 P1-3：聊天区重构为主任务流

核心需求：

- 明确区分欢迎引导、会话、排队状态、输入区和日志区。
- 保持历史稳定，不再清空主对话感知。
- 让系统反馈文案和页面动作语义一致。

价值：

- 强化“主 Agent 是唯一长期主体”的心智。
- 降低页面切换和信息丢失感。

### 需求 P1-4：统一视觉与交互系统

核心需求：

- 建立设计变量层。
- 收敛颜色角色和按钮状态。
- 去掉大面积终端风格和 emoji 语义依赖。
- 补全响应式与可访问性基础。

价值：

- 从“可演示”提升到“可稳定使用”。
- 增强产品感和可信度。

## 5.3 P2：构建中期产品护城河

### 需求 P2-1：求职策略层

方向：

- 让系统管理岗位篮子、城市偏好、薪资带、投递节奏和优先级，而不只是搜索结果。

### 需求 P2-2：机会评分与推荐理由

方向：

- 对职位输出可解释的匹配分、风险点、推荐动作和优先级建议。

### 需求 P2-3：投递后管理

方向：

- 增加跟进提醒、状态时间线、拒信归因、复盘记录和邮件摘要。

### 需求 P2-4：个人成长闭环

方向：

- 串联简历评价、JD 差距分析、面试模拟和改进建议，形成学习型反馈系统。

### 需求 P2-5：可解释自动化

方向：

- 用户不仅知道 Agent 做了什么，还知道它为什么这么做、下一步准备做什么、在哪些节点需要人工授权。

## 6. 建议的阶段目标

### 阶段一：先把入口和闭环站住

范围：

- 首次配置引导。
- 统一任务状态。
- 统一结果落点。
- 能力降级面板。
- 职位原子操作接口。

阶段目标：

- 用户第一次使用不迷路。
- 用户发起任务后不迷失。
- 用户开始相信系统的数据和反馈。

### 阶段二：把高频效率做出来

范围：

- 职位处理台。
- 简历工作流。
- 聊天区重构。
- 视觉和交互收敛。

阶段目标：

- 用户可以高频、稳定地使用这套系统处理真实求职任务。

### 阶段三：把自动化做成护城河

范围：

- 求职策略层。
- 投递后管理。
- 成长闭环。
- 可解释自动化。

阶段目标：

- 产品从“工具集合”升级为“持续协作的求职系统”。

## 7. 产品验收指标建议

### 7.1 北极星方向

- 用户每周主动使用天数。
- 每周被处理的职位数量。
- 每周完成的有效求职动作数量。

### 7.2 阶段一重点指标

- 首次启动后完成基础配置的转化率。
- 首次访问到第一次成功搜索的完成率。
- 长任务失败后用户仍继续下一步操作的比例。
- 因状态不清或结果不可见而触发的重复操作率。

### 7.3 阶段二重点指标

- 单次职位处理平均耗时。
- 简历任务完成率。
- 批量操作使用率。
- 职位筛选和搜索功能使用率。

## 8. 一句话结论

JobClaw 下一阶段最该做的，不是继续堆更多 Agent 能力，而是先把“可信的任务闭环”和“高频求职处理效率”做出来。只有这两个基础成立，自动化、投递和策略能力才会真正转化成产品价值。
